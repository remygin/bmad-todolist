# PRD Quality Review — BMAD Todo List: Параметры пользователей на карточках

## Overall verdict

Addendum написан дисциплинированно: дельта зафиксирована с явными границами, acceptance criteria testable,
бизнес-правила формализованы. Основной риск — дублирование сборки CardResponse в двух сервисах
(RR-1/OQ-1) без обязательного требования синхронизации; второй значимый риск — некорректная
рекомендация по разграничению null/absent для assigneeId в update-запросе (OQ-4). Оба риска
зафиксированы как открытые вопросы, что корректно для addendum, но снижают decision-readiness
до adequate.

## Decision-readiness — adequate

Addendum корректно разграничивает принятые решения (BR-1–BR-8, дельта FR-11/FR-12, изменения
Non-Goals) и открытые вопросы (OQ-1–OQ-6). Торговые компромиссы явно названы в OQ.
Однако OQ-4 содержит рекомендацию (вариант D: @JsonInclude(NON_NULL)), которая технически
не решает заявленную проблему — Jackson не различает отсутствующее поле и явный null для Long.
Это снижает доверие к рекомендациям addendum.

### Findings
- **medium** OQ-4: неполнота рекомендации по разграничению null/absent (§9 OQ-4) — Рекомендация D не работает с дефолтной Jackson-десериализацией для Long. *Fix:* Явно указать ограничение варианта D; рекомендовать Optional&lt;Long&gt; или кастомный десериализатор.
- **medium** RR-1/OQ-1: риск рассинхронизации не митигирован на уровне требований (§4, §8, §11) — RR-1 высокой вероятности не имеет обязательного требования синхронизировать сборку CardResponse. *Fix:* Добавить обязательное требование: «CardResponse собирается единым методом; BoardService делегирует CardService».

## Substance over theater — strong

Addendum лишён «театральных» секций. Каждый раздел (дельта FR, бизнес-правила, AC, edge cases,
ограничения, out of scope, affected areas, OQ, assumptions, regression risks) — содержательный
и обоснованный. Никаких шаблонных фраз, инновационной риторики или раздутых персон.

### Findings
Нет значимых замечаний.

## Strategic coherence — adequate

Тезис addendum ясен: расширить модель карточки полями creator/assignee. FR, бизнес-правила и
ограничения следуют из этого тезиса. Связь с существующей архитектурой (User entity, Board.author,
single-user model) прослеживается в assumptions §10. Однако cross-board assignee (EC-4) обсуждается
в PIA §5 как требующий политики, но в addendum политика раскрыта минимально (BR-5).

### Findings
- **low** EC-4/PIA: политика cross-board assignee не полностью раскрыта (§5 EC-4, PIA §5) — BR-5 утверждает отсутствие прав доступа у assignee, но не документирует, валидируется ли принадлежность к доске. *Fix:* Явно указать: «assignee может быть любым ADMIN в системе, независимо от владения доской. Права доступа не предоставляются.»

## Done-ness clarity — strong

Каждый AC (AC-1–AC-8) testable: Given/When/Then с конкретными ожиданиями. Edge cases (EC-1–EC-8)
покрывают corner cases. BR-1–BR-8 однозначны. Consequences FR-11/FR-12 содержат проверяемые
условия. Нет размытых формулировок («удобный интерфейс», «хорошая производительность»).

### Findings
- **low** AC-8: acceptance criteria для отображения на фронтенде не специфицирует формат (§4 AC-8) — указан факт отображения username, но не расположение на карточке. *Fix:* Добавить минимальные требования к расположению (под заголовком, мелким шрифтом).

## Scope honesty — strong

Out of scope (§7) явный и подробный (11 пунктов). Assumptions (§10) — 8 предположений, все
обоснованы ссылками на существующую архитектуру. Regression risks (§11, RR-1–RR-4) с указанием
вероятности и митигации. Non-Goals базового PRD скорректированы в §2.3 addendum — честная
фиксация изменения scope.

### Findings
- **low** Non-Goals: механизм применения дельты к базовому PRD не определён (§2.3) — addendum изменяет Non-Goals §5 базового PRD, но не уточняет, будет ли base PRD отредактирован. *Fix:* Зафиксировать: «При утверждении base PRD получает новую редакцию с изменениями из §2.3, status: updated.»

## Downstream usability — adequate

Глоссарий расширен (§1) терминами Creator и Assignee — закреплены в FR и AC. FR ID (FR-11, FR-12)
сохранены из базового PRD. BR, AC, EC, OQ имеют уникальные ID. Cross-ссылки на базовый PRD
(§5, FR-11, FR-12, UJ-2) корректны. Однако OQ-2 не специфицирует требования безопасности для
нового эндпоинта, что может затруднить downstream-шаг архитектуры.

### Findings
- **medium** OQ-2: отсутствие требований безопасности для GET /api/users (§8, §9 OQ-2) — не указано, требуется ли аутентификация ADMIN для нового эндпоинта. *Fix:* Добавить: «GET /api/users требует аутентификации ADMIN; неаутентифицированные запросы — 401/403.»

## Shape fit — strong

Формат «on_addendum» адекватен для данного изменения: feature затрагивает множественные слои,
требует миграции данных и несёт regression risk. Addendum правильно позиционирован как дельта
к базовому PRD (status: final). Документ самодостаточен для согласования scope, но оставляет
архитектурные решения downstream-шагу — корректное разграничение ответственности.

### Findings
Нет значимых замечаний.
