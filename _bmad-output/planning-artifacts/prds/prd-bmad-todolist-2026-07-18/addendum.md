# PRD Addendum: Назначение исполнителя и создателя на карточку

## 1. Обоснование

Настоящий addendum расширяет brownfield-PRD «BMAD Todo List — Kanban с JWT-админом» (статус: final, 2026-07-18). Фича «Назначение исполнителя и создателя на карточку» вводит базовую коллаборацию на Доске: каждая Карточка получает **создателя** (обязательно) и **исполнителя** (опционально). Это первый шаг к многопользовательскому режиму: пользователи могут видеть, кто создал задачу и на кого она назначена, а исполнитель автоматически получает доступ к Доске.

**Характер изменения:** фича отменяет граничное условие из §5 Non-Goals существующего PRD (п. «Нет ... назначения исполнителей на Карточках»), меняет фундаментальное допущение о single-user системе (Глоссарий, §2.2) и расширяет контракт FR-5 (список Досок). Решение принято в `prd-format-decision` (2026-07-23): формат — Addendum, не inline-редакция.

## 2. Изменения в Глоссарии

Следующие термины **добавляются** в Глоссарий родительского PRD:

| Термин | Определение |
|--------|-------------|
| **Создатель (Creator)** | Пользователь, создавший Карточку. Фиксируется при создании и не меняется. NOT NULL — каждая Карточка всегда имеет Создателя. |
| **Исполнитель (Assignee)** | Пользователь, на которого назначена Карточка. Опционально (nullable). Назначение автоматически даёт Исполнителю доступ к Доске. |
| **Исполнитель (роль, не-ADMIN)** | Пользователь без роли `ADMIN`, который может просматривать Доску и Карточки, где назначен Исполнителем. Не может создавать/редактировать/удалять Карточки и управлять Доской. |

## 4. Приёмочные критерии (AC)

### 4.1 Backend

| ID | Критерий |
|----|----------|
| AC-1.1 | `POST /api/cards` без `assigneeId` создаёт Карточку с `creator = текущий пользователь`, `assignee = null` |
| AC-1.2 | `POST /api/cards` с валидным `assigneeId` создаёт Карточку с назначенным Исполнителем; Исполнитель получает доступ к Доске |
| AC-1.3 | `POST /api/cards` с несуществующим `assigneeId` → 404 |
| AC-2.1 | `PUT /api/cards/{id}` с `assigneeId: null` снимает назначение (если был Исполнитель) |
| AC-2.2 | `PUT /api/cards/{id}` с новым `assigneeId` меняет Исполнителя; новый Исполнитель получает доступ к Доске |
| AC-3.1 | `GET /api/boards/{id}` для пользователя-Исполнителя возвращает Доску (если есть хотя бы одна Карточка с его `assignee_id`) |
| AC-3.2 | `GET /api/boards/{id}` для пользователя без доступа → 404 |
| AC-3.3 | `GET /api/boards` возвращает Доски, где пользователь автор ИЛИ Исполнитель |
| AC-4.1 | Проверка доступа использует EXISTS-подзапрос к `cards.assignee_id` (без отдельной таблицы `board_members`) |

### 4.2 Frontend

| ID | Критерий |
|----|----------|
| AC-5.1 | Карточка отображает username Создателя (read-only) |
| AC-5.2 | Карточка отображает аватар с инициалами Исполнителя (или placeholder, если не назначен) |
| AC-5.3 | Форма создания/редактирования Карточки содержит dropdown выбора Исполнителя |
| AC-5.4 | Dropdown загружает список пользователей через `GET /api/users` |
| AC-6.1 | Исполнитель (не ADMIN) видит Dashboard со своими Досками |
| AC-6.2 | Исполнитель видит Доску целиком (все Карточки, включая чужие) |
| AC-6.3 | Исполнитель НЕ видит кнопки/формы создания/редактирования/удаления Карточек |
| AC-6.4 | Исполнитель НЕ видит UI управления Доской/колонками |

## 5. Edge Cases (EC)

| ID | Сценарий | Ожидаемое поведение |
|----|----------|---------------------|
| EC-1 | ADMIN создаёт Карточку и назначает себя Исполнителем | Корректно: creator = assignee = ADMIN |
| EC-2 | ADMIN снимает Исполнителя с Карточки, но пользователь остаётся Исполнителем на других Карточках той же Доски | Доступ сохраняется |
| EC-3 | Два ADMIN (bootstrap + ещё один) назначают друг друга Исполнителями | Согласовано с моделью (ADMIN имеет все права) |
| EC-4 | Пользователь назначен Исполнителем, но его аккаунт удалён | Зависит от стратегии удаления (OQ-4) |
| EC-5 | Массовое назначение: ADMIN создаёт 100 Карточек одному Исполнителю | Нет ограничения на число Карточек per assignee — допустимо |
| EC-6 | Попытка назначить несуществующего пользователя | 404 |
| EC-7 | Попытка снять assignee через передачу `assigneeId: null` | Работает (AC-2.1) |
| EC-8 | Повторное назначение того же пользователя | Идемпотентно (BP-10) |

## 6. Ограничения (Constraints)

| ID | Ограничение | Комментарий |
|----|-------------|-------------|
| C-1 | Нет self-signup / регистрации | Пользователи должны быть созданы до фичи — механизм не определён (OQ-5) |
| C-2 | Нет ролевой модели кроме ADMIN | Исполнитель не имеет роли — доступ только через `assignee_id` |
| C-3 | Нет отдельной таблицы `board_members` | Доступ определяется EXISTS-подзапросом |
| C-4 | `@PreAuthorize("hasRole('ADMIN')")` остаётся на Kanban-контроллерах | Исполнитель не сможет вызывать мутации |
| C-5 | UI на React без глобального store | Усложняет управление состоянием доступа для разных ролей |
| C-6 | Нет кэширования прав | Каждый запрос проверяет EXISTS — нагрузка на БД |
| C-7 | `open-in-view: false` | Связи creator/assignee нужно загружать внутри `@Transactional` |
| C-8 | Аватар — только инициалы (первые буквы username) | Без gravatar/загрузки фото |

## 7. Out of Scope (явно не входит в фичу)

- Механизм создания дополнительных пользователей (endpoint `POST /api/users` или SQL) — требуется как precondition, но не реализуется в рамках этой фичи
- Уведомления о назначении (in-app / email / push)
- Self-signup / регистрация / восстановление пароля
- Мульти-тенантность, команды, workspace
- Отдельная таблица `board_members` — доступ через EXISTS
- Кастомные статусы, вложения, комментарии, метки, сроки, напоминания
- Публичный API, вебхуки, сторонние интеграции
- Gravatar / загрузка аватаров
- Мобильные нативные приложения
- Экспорт / импорт / восстановление данных

## 9. Открытые технические вопросы (без проектирования решения)

### OQ-1: Доступ Исполнителя к Доске — вся Доска целиком?

Исполнитель видит Доску целиком (все Карточки) — упрощает реализацию, даёт контекст.
**Альтернатива:** только свои Карточки — больше приватности.
**Статус:** требуется решение PO.

### OQ-2: Что происходит с доступом Исполнителя при снятии с последней Карточки?

- Доступ теряется немедленно (проверка EXISTS на каждый запрос)
- Доступ сохраняется до конца сессии (кэширование)
- Доступ сохраняется навсегда (пользователь остаётся участником Доски)
**Статус:** требуется решение PO.

### OQ-3: Как Исполнитель выбирается в UI?

- Выпадающий список всех пользователей системы (через `GET /api/users`)
- Текстовое поле с автодополнением (поиск на сервере)
- Список пользователей, уже имеющих доступ к данной Доске
**Статус:** требуется решение PO.

### OQ-4: Стратегия удаления пользователя, назначенного Исполнителем?

- Запретить удаление (пока есть назначения)
- Разрешить удаление, сбросить `assignee_id = NULL`
- Каскадное удаление
**Статус:** требуется решение PO / архитектурное решение.

### OQ-5: Как создаются дополнительные пользователи (не bootstrap-админ)?

- Добавить `POST /api/users` (только ADMIN)
- Через SQL-скрипт / миграцию
- Оставить за рамками фичи; указать в документации, что пользователи существуют заранее
**Статус:** precondition — требуется решение до реализации.

### OQ-6: API `GET /api/users` — какие поля возвращать?

- Только `id` + `username`
- `id` + `username` + `role`
- Полный профиль (без пароля)
**Статус:** требуется архитектурное решение.

### OQ-7: Пагинация `GET /api/users`?

При 2–10 пользователях не нужна, но закладывать ли?
**Статус:** требуется архитектурное решение.

### OQ-8: Как FE определяет, что текущий пользователь — ADMIN vs Исполнитель?

- Через роли из JWT (`ADMIN` vs отсутствие роли)
- Через отдельный флаг в `BoardResponse`
- Через отдельный endpoint
**Статус:** требуется архитектурное решение.

### OQ-9: Связи `creator`/`assignee` — стратегия загрузки?

LAZY + `@EntityGraph` / `JOIN FETCH` в CardService.
**Статус:** архитектурное решение.

### OQ-10: Затрагивает ли move-операция (`PATCH /api/cards/{id}/move`) assignee?

По текущему описанию — нет. Требуется подтверждение.
**Статус:** требуется подтверждение.

## 10. Риски (Risks)

| ID | Риск | Описание | Mitigation |
|----|------|----------|------------|
| R-1 | Утечка контекста Доски | ADMIN случайно назначает не того пользователя | Рассмотреть confirm при назначении нового пользователя |
| R-2 | Нарушение приватности | Исполнитель видит все Карточки, включая конфиденциальные | Зависит от решения OQ-1 |
| R-3 | Назначение без согласия | Пользователь может быть назначен без его ведома | Нет защиты в MVP |
| R-4 | Потеря доступа mid-session | Последняя Карточка переназначена — доступ теряется мгновенно | Рассмотреть предупреждение при снятии |
| R-5 | Нет уведомления о назначении | Исполнитель не знает, что назначен | Out of scope для MVP |
| R-6 | Поломка существующих тестов | V3 миграция и NOT NULL creator_id требуют обновления тестовых данных | Обновить fixtures в рамках реализации |

## 11. Неизменяемые инварианты (подтверждение)

Следующие аспекты родительского PRD **не меняются** данной фичей:

- Инвариант трёх системных статусов (`TODO`/`IN_PROGRESS`/`DONE`) — остаётся
- Владение Доской (`author_id`) — автор всегда имеет доступ
- CRUD Досок, настройка Колонок — без изменений
- Механизм аутентификации (JWT Bearer, sessionStorage)
- Единственная роль ADMIN на уровне `@PreAuthorize`
- Нормализация позиций Карточек после мутаций
- Формат ошибок (`message` + `details`)
- FE-стек (React/TS/Vite/@dnd-kit)
- Аватар — только инициалы (без gravatar/фото)
- Механизм выхода, health check, политика CSRF/session
